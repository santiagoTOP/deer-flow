## 问题诊断

`stream-bridge-deep-dive.md` 的 3.3 节（行 490-832，约 342 行）违反了本文档自己建立的教学范式：

- 文档**开头**的"第 0 步"（0.1~0.9）用了大量生活比喻（电话通话、切菜做饭、餐厅等位、游戏存档）把 SSE / async / checkpoint / lease 等概念逐个铺好，才进入代码。

新增铺垫小节包含 6 个概念（每个用一个生活比喻 + 大白话，约半个屏幕篇幅）：

1. **IM 机器人 / 通道（Channel）是什么**
   - 比喻：飞书/Slack 里的机器人就像银行大堂的"迎宾机器人"，它替你把问题转给后台专家（agent），再把专家的回答转给你。
   - 关键点：机器人本身不会思考，它只是个"传话筒"。

2. **长连接 vs webhook（平台怎么把消息送到我们这）**
   - 比喻：收快递两种方式——① 你主动去快递柜轮询（long-polling/WebSocket，飞书/Telegram/钉钉）；② 快递员直接按门铃送到家（webhook，需要公网 IP，GitHub 用这个）。
   - 点出：DeerFlow 大部分 IM 走长连接，所以内网也能部署。

3. **MessageBus —— 进程内的"取号机"**
   - 比喻：医院取号机。患者（入站消息）取号排队（`asyncio.Queue`），叫号后护士（ChannelManager）处理；处理完医生写处方（OutboundMessage），广播给对应科室。
   - 关键点：只活在 Gateway 这一个进程里，不跨网络。

4. **StreamBridge —— 跨网络边界的事件日志**（呼应第 0 步已讲的，一句话带过 + 强调它跨 HTTP）
   - 比喻：前面讲过的"每个 run 一份存档"。这里只补一句：它和 MessageBus 是**两套完全独立**的系统，名字像但互不 import——这是后面最大的坑。

5. **ChannelManager —— 身兼两职的"翻译员"**
   - 比喻：一个会两种语言的翻译员，左耳听 IM（MessageBus），右嘴说给 agent 听；它对 agent 说话的方式是**发 HTTP 请求**（自回环）。
   - 关键点：这就是"IM 怎么接进 agent runtime"的核心——靠一个既是 IM 协调者、又是 HTTP 客户端的角色。

6. **HTTP 自回环（self-loopback）—— 为什么不直接调函数**
   - 比喻：餐厅经理明明在餐厅里，点菜却偏要用手机在外卖 App 下单——这样后厨的接单系统、记账系统、排队系统全都一样跑，不用为"老板亲自点的单"单独开一套。
   - 关键点：鉴权 / CSRF / 状态机 / 线程忙保护，浏览器能享受的 IM 自动也享受。

小节末尾用一段话承上启下：**"现在带着这些概念，再看下面那张全景图，每个框就能对上号了。"** —— 完成"由知识（铺垫）到实践（看真实架构图 + 逐阶段代码）"的桥接。

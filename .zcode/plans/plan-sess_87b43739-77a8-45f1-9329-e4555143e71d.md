## 核心论点（一句话立意）

> `StreamBridge` 把**事件流的生命周期**和 HTTP 解耦；`RunManager` 把**执行 + 状态机的生命周期**和 HTTP 解耦。

一篇 HTTP 请求创建的 run，要活得比这次请求久，而且会被**后来的别的请求**观察、取消、重新生成、重启时恢复——朴素"路由里 `asyncio.create_task(agent.run())` 就不管"做不到。这是 `RunManager` 存在的全部理由。

### 第 1 步：朴素写法长什么样
虚构但真实的"如果没有 RunManager"路由写法：
```python
async def stream_run_naive(thread_id, body, request):
    agent = build_agent(...)
    task = asyncio.create_task(agent.astream(...))
    return StreamingResponse(consume(task), ...)
```
逐行解释，点出根本问题：**run 生命周期被焊死在路由局部变量里——响应一返回，没有任何东西记得这个 run。** 6 个麻烦全从此派生。

### 第 2 步：6 个麻烦逐一展开（每个"场景 → 朴素为何做不到 → RunManager 怎么解决"）

**麻烦 1：第二次请求想看这个 run（列表/详情/join）**
- `_runs: dict[run_id, RunRecord]` 注册表 + `_runs_by_thread` 二级索引（自讲 `manager.py:217-245` docstring 里的"锁内同步"不变量）
- `get_run_manager(request)` 工厂代码（`deps.py:351-365`）—— 从 `app.state.run_manager` 取单例 + 503 兜底
- **RunRecord 字段表本篇自包含**（贴 `manager.py:148-184` 全字段，分组注释：标识/状态/执行/多 worker 归属/token 计量）—— 不再"引用 07"

**麻烦 2：进程重启后 run 历史不能丢**
- 双层架构 `RunManager`（内存快）+ `RunStore`（持久化不丢）
- `_record_from_store` 的 `store_only=True` 标志（`manager.py:378-414`）—— 与 stream-bridge §4.1 呼应但本篇讲透**它怎么来的**
- 装配代码 `deps.py:271-312`（RunRepository vs MemoryRunStore 二选一）

**麻烦 3：同 thread 不能并发跑两个 run（多任务策略）**
- `create_or_reject` 三步原子操作逐行（`manager.py:920-1086`）：①本地 inflight 检查 ②**持本地锁的同时写 store**（store 是跨进程真相之源）③store 成功后才登记本地
- "锁序不变量"docstring（938-944）+ store 层部分唯一索引 `uq_runs_thread_active`（`persistence/run/model.py:48-57`）
- 三种策略 reject/interrupt/rollback 语义

**麻烦 4：多 worker 下 run 归谁所有（lease + heartbeat）**
- lease 数据：`owner_worker_id` + `lease_expires_at`（RunRecord 字段 + DB 列，引用 migration 0004）
- 续租 `_renew_leases`（`manager.py:1266-1322`）逐行 —— `update_lease` WHERE 守卫（只续自己的、还活着的），续租失败说明被接管 → 主动 `task.cancel()`
- **心跳循环 `_heartbeat_loop`（1222-1264）本篇自贴代码 + 逐行** —— 为什么续租间隔 `lease_seconds // 3`，为什么每 3 周期才对账；关键取舍：**心跳挂了 = 所有 run 被当孤儿接管** → 每个操作 try/except 包死
- 多 worker 启动守卫 `deps.py:49-85`

**麻烦 5：worker 崩了，run 谁收尸（孤儿恢复 + 接管）**
- 启动对账 `reconcile_orphaned_inflight_runs`（`manager.py:1088-1144`）+ `deps.py:313-327` —— 扫 lease 过期的 pending/running 标 error，`bridge.publish_end` 发结束信号（与 stream-bridge §4.6 衔接，引用不重复）
- 运行时被动接管 `cancel` 非本地路径（`manager.py:840-903`）逐行 —— `is_lease_expired` → `claim_for_takeover` 条件 UPDATE（WHERE lease 已过期）→ `CancelOutcome.taken_over`；重点讲**为什么用条件 UPDATE 而非"先读后写"**（关闭竞态窗口）
- HTTP 层 `cancel_run`（`thread_runs.py:573-615`）：taken_over→200，lease_valid_elsewhere→409+Retry-After

**麻烦 6：优雅停机 shutdown —— run 不能跟着进程暴毙**
- `shutdown(timeout=5.0)`（`manager.py:1339-1432`）逐行 —— 停心跳、给所有 inflight 发 abort+cancel、**有界等待** timeout、没自己结束的标 interrupted 落库
- 引用 issue #3373（为什么要在 checkpointer 连接池关闭**之前** drain）
- `_drain_inflight_runs`（`deps.py:88-111`）调用顺序

### 第 3 步：把 6 个麻烦的本质串起来
一张表对比"HTTP 请求生命周期" vs "run 执行生命周期"，论证 RunManager 是把两者解耦的中间层，与 stream-bridge 结尾结论形成**双璧**：
- StreamBridge 解耦**事件流**
- RunManager 解耦**执行 + 状态**

**3. 入站链路（IM → agent），逐阶段逐行**
分阶段（参考 3.4.1 的分阶段写法）：

- **阶段 A：平台消息到达通道**——讲清只有 GitHub 走 FastAPI webhook（`POST /api/webhooks/github`，`github_webhooks.py:172`，HMAC 验签 `_verify_signature:101`）；Feishu/Slack/Telegram 走平台长连接（WebSocket / Socket Mode / long-poll），不需要公网 webhook。以飞书为例：lark WS 线程 → `_on_message`（`feishu.py:954`）→ `_schedule_prepare_inbound`（跨线程 marshal 回主循环）。
- **阶段 B：通道发入站消息到 MessageBus**——`bus.publish_inbound(inbound)`（飞书 `feishu.py:917`）。贴 `InboundMessage` 关键字段。
- **阶段 C：ChannelManager 消费入站**——`_dispatch_loop`（`manager.py:1126`）`await bus.get_inbound()` → 去重 → `asyncio.create_task(_handle_message)`，并发信号量 `max_concurrency=5`。
- **阶段 D：解析线程 + 运行参数**——`_handle_chat`（`manager.py:1504`）get/create thread（`client.threads.create`）→ `_handle_chat_on_thread`（`manager.py:1563`）→ `_resolve_run_params`（assistant_id / run_config / run_context）。
- **阶段 E：三路分流**——贴 `_handle_chat_on_thread:1604-1663` 的核心分支代码（streaming / fire_and_forget / wait），逐行解释。
- **阶段 F：langgraph_sdk 内部 = 一次 HTTP POST**——贴 `_get_client`（`manager.py:1081-1094`）：`url=self._langgraph_url`（`http://localhost:8001/api`）、`create_internal_auth_headers()`、CSRF 双提交。说明虽然 ChannelManager 跑在 Gateway 同进程，但**走 HTTP 自回环**命中 `/api/threads/{tid}/runs/stream` → `stream_run`（`thread_runs.py:496`）→ `start_run`（`services.py:608`）→ `asyncio.create_task(run_agent(...))`。**这就是 IM 接入 agent runtime 的物理接合点。**
- 强调：到 `start_run` 这一步，IM 发起的 run 和浏览器发的 run、调度器发起的 run 汇聚到**同一条 agent 执行管道**。

**4. 出站链路（agent → IM），逐阶段逐行**

- **阶段 G：agent worker 往 StreamBridge publish**——`run_agent`（`worker.py:246`）只调 `bridge.publish(run_id, event, data)`（如 `worker.py:350/502/523`），**从不碰 MessageBus**。
- **阶段 H：ChannelManager 作为 HTTP 客户端从 StreamBridge 消费**——`client.runs.stream` 在 SDK 内部解析 SSE，把 chunk 还给 `_handle_streaming_chat`（`manager.py:1712`）的 `async for chunk`。说明这一步就是 3.1/3.6 讲的 `sse_consumer → bridge.subscribe` 的另一端。
- **阶段 I：manager 把 chunk 转成 OutboundMessage 发到 MessageBus**——贴 `_handle_streaming_chat:1776` 的 `bus.publish_outbound(OutboundMessage(is_final=False, ...))`，以及 finally 块 `manager.py:1826` 的 `is_final=True`。说明节流（`STREAM_UPDATE_MIN_INTERVAL_SECONDS=1.0` / `STREAM_UPDATE_MIN_CHARS=60`）和 "▉" 游标。
- **阶段 J：MessageBus 扇出到对应通道**——贴 `publish_outbound:186-190` 的 for 循环；贴 `Channel._on_outbound`（`base.py:158-168`）的 `if msg.channel_name == self.name` 名字门。
- **阶段 K：通道调平台 API 把消息送回用户**——飞书为例：`FeishuChannel.send`（`feishu.py:273`）→ `_send_card_message`（`feishu.py:607`）→ `asyncio.to_thread(self._api_client.im.v1.message.patch, request)`（`feishu.py:526`）；`is_final` 时 pop running card + 加 "DONE" reaction。

**5. 回到 3.3 的老问题：IM 到底是不是订阅者？**
- 收束结论：**是**，但是"远程的、经 HTTP 自回环进来的"StreamBridge 订阅者，和浏览器标签页本质相同。
- 同时强调：IM 在 ChannelManager 侧的内部协调（IM↔manager）走的是 **MessageBus**，不是 StreamBridge——这是两层不同的总线，别混。

**6. 三个核心设计点回顾**（仿 3.4.1 末尾）
1. **两条总线职责分离**：StreamBridge 管跨 HTTP 边界的 run 事件流；MessageBus 管进程内 IM↔manager 的整条消息往返。两者唯一的"粘合"是 ChannelManager 本身是 Gateway 的 HTTP 客户端。
2. **HTTP 自回环复用**：ChannelManager 不直接调 agent，而是经 `langgraph_sdk` POST 自家 `/runs/*`，复用 `start_run` 全套管道（鉴权、context、状态机），和浏览器/调度器走同一条路。
3. **能力分流按通道特性**：streaming/.wait/create 三条路径对应不同 IM 平台的交互范式（实时打字 vs 跑完回帖 vs agent 自主回帖）。
